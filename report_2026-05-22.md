# OmegaETH Sequencer — Changes Since Last Push
**Date:** 2026-05-22  
**Last commit:** `760de8b` (added report)

## Overview

This document describes all changes made to the sequencer after the last git push. The changes fall into five categories: simulation architecture rewrite, ordering latency fix, wait_score fix, slot boundary inclusion time, and DES verbose scoring output.

---

## 1. Simulation Architecture: Fixed 60s Window → Dual-Trigger Event-Driven Loop

**File:** `sequencer/src/main.rs`

### Before
The simulation used a fixed 60-second tick loop. Every iteration admitted all txs that arrived in the past 60 seconds, ordered them, packed all blobs, and submitted everything. `sim_clock_ms` advanced by exactly `batch_timeout_ms` (60,000ms) per tick regardless of actual tx arrival times.

**Problems:**
- Unrealistic: real sequencers do not operate on fixed windows
- `wait_score` in DES was a tiebreaker only within one window — it never grew across windows
- All blobs in a window were drained at once; no individual blob submission logic

### After
Replaced with a dual-trigger event-driven loop:

**Capacity trigger:** fires when `mempool_size >= 128KB`. Submits **one blob only**, leaves remaining txs in the mempool, keeps the timer running. This is how real sequencers behave — submit as soon as one blob is ready, do not wait for the time limit.

**Time trigger:** fires when `batch_timeout_ms` has elapsed since the first tx arrived in the current batch. Submits all remaining txs as partial blobs, then resets the timer.

**Event loop mechanics:** `sim_clock_ms` advances to the earlier of (next tx arrival, time trigger deadline) rather than by a fixed amount. This means the simulation moves at the pace of the actual dataset timestamps.

**`batch_timeout_ms` changed from 60,000ms to 600,000ms (10 minutes).** The time trigger is now a safety net. In high-throughput workloads like `large_heavy`, the capacity trigger dominates and the time trigger rarely fires.

## 2. Ordering Latency: Approximation → Exact Measurement

**File:** `sequencer/src/metrics.rs`

### Before
`record_ordering_latencies` spread selection times evenly across the batch window:
```rust
let selected_at_ms = window_start_ms + (idx as u64 * batch_timeout_ms / n as u64);
let latency = selected_at_ms.saturating_sub(tx.arrival_ms);
```
This was a fake approximation — no tx was actually selected at those times.

### After
Ordering latency is now exact:
```rust
let latency = seal_time_ms.saturating_sub(tx.arrival_ms);
```
`seal_time_ms` is `sim_clock_ms` at the exact moment the trigger fired and the ordering algorithm ran. Every tx in a blob shares the same seal time, which is correct — they were all selected in the same ordering call.

## 3. wait_score: Linear Normalization → Exponential Aging

**File:** `sequencer/src/ordering.rs`

### Before
```rust
let wait_score = (wait_time / self.batch_timeout_ms).min(1.0);
```
With `batch_timeout_ms = 600,000ms`, a tx waiting 4 seconds got `wait_score = 0.0067` — effectively zero. Alpha (the starvation prevention weight) was mathematically dead in high-throughput scenarios where the capacity trigger fires every few seconds.

### After
```rust
let lambda = 0.0001_f64;
let wait_score = 1.0 - (-lambda * wait_time).exp();
```
Exponential aging gives meaningful scores immediately:
- 4s wait → score ≈ 0.33
- 10s wait → score ≈ 0.63
- 60s wait → score ≈ 1.00

This also removes the dependence on `batch_timeout_ms` as a normalizer. The `batch_timeout_ms` field was removed from the `DesOrdering` struct entirely — it is now only used by the time trigger in `main.rs`.

## 4. Inclusion Latency: Real Wall-Clock → Simulated Slot Boundary

**File:** `sequencer/src/main.rs`

### Before
Inclusion latency came from the real wall-clock time of `get_receipt()` in `blob_sender.rs`:
```rust
Ok((tx_hash, inclusion_ms)) => metrics.record_inclusion(inclusion_ms);
```
This was real elapsed time waiting for the devnet to confirm the blob — ~6 seconds per blob, but noisy and dependent on network conditions.

### After
Inclusion latency is calculated from the simulation clock using slot boundary math:
```rust
let slot_ms: u64 = 6_000; // matches network_params.yaml: seconds_per_slot: 6
let t_inclusion = ((sim_clock_ms + slot_ms) / slot_ms) * slot_ms;
metrics.record_inclusion(t_inclusion - sim_clock_ms);
sim_clock_ms = t_inclusion;
```
This snaps `sim_clock_ms` to the next 6-second slot boundary and records the gap as inclusion latency (1–6s). The devnet's `send_blob` call still fires for real blob submission, but its wall-clock timing is discarded.

**Capacity trigger:** each blob lands on the next slot boundary. `sim_clock_ms` advances, and any txs that arrived during the slot window are admitted immediately afterward.

**Time trigger:** all blobs submitted in one time-trigger firing land in the **same slot** (calculated once before the loop). `sim_clock_ms` advances once after all blobs are submitted, not once per blob.

**Why this matters:** the slot boundary advancement accumulates across the simulation. After 171 blobs × up to 6s per slot = up to 1,026s of additional sim time on top of the 600s dataset. This correctly models the fact that a real sequencer waiting for L1 confirmation after each blob delays admission of subsequent txs.

## 5. DES Verbose Scoring Output

**File:** `sequencer/src/ordering.rs`

Added `DES_VERBOSE=1` environment variable. When enabled, prints per-tx scores for every tx selected by the DES algorithm:

```
blob=  0 tx=    10 type=other    size= 2116B  wait=0.109 compress=0.788 fit=0.016 des=0.3016
blob=  0 tx=    87 type=other    size=10980B  wait=0.332 compress=0.100 fit=0.929 des=0.4584
blob=  1 tx=     5 type=other    size= 9725B  wait=0.131 compress=0.100 fit=0.074 des=0.1015
```

The blob number is now **global** across the entire simulation run (0 through N-1), not local to each ordering call. This required adding a `blob_offset: usize` parameter to the `OrderingPolicy::order()` trait method. `main.rs` passes `submission_count` as the offset so blob numbers count up correctly across all capacity trigger firings.

Usage:
```bash
DATASET="../large_heavy.csv" ORDERING_POLICY="des" RUST_LOG="error" DES_VERBOSE=1 cargo run --release > des_verbose.txt
```

## 6. Experimental Results (large_heavy.csv, 2026-05-22)

Run on Kurtosis Reth+Lighthouse devnet (enclave: `gloomy-ravine`), 6s slots, Deneb enabled from epoch 0.

| Metric | FCFS | DES | Change |
|--------|------|-----|--------|
| Total Blobs | 190 | 171 | -10.0% |
| Avg Uncompressed Fill Rate | 89.47% | 99.41% | +9.94pp |
| Avg Compressed Fill Rate | 82.29% | 91.42% | +9.13pp |
| Avg Compression Ratio | 1.09:1 | 1.09:1 | — |
| Ordering Avg | 138,912ms | 134,817ms | -3.0% |
| Ordering P95 | 258,930ms | 504,957ms | +95% |
| Ordering Max | 272,601ms | 589,894ms | +116% |
| Inclusion Avg | 5,147ms | 5,235ms | ~same |
| Inclusion Max | 5,999ms | 5,999ms | — |

**Key findings:**
- DES saves 19 blobs (10.0%) with near-perfect fill rate (99.41%) on `large_heavy`
- DES tail latency is ~2× worse at P95 and max due to HOL-blocking of large txs
- DES average latency is slightly lower because fewer blobs means fewer slot boundary advancements and lower average sim time per tx
- Inclusion latency is bounded by the 6s slot — max 5,999ms confirms slot boundary math is correct
- Compression ratio is identical (1.09:1) for both policies on this workload — large calldata compresses poorly regardless of ordering
