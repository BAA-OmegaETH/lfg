# OmegaETH Sequencer — Project Documentation
**Date:** 2026-05-09

---

## 1. Project Overview

OmegaETH is a custom Layer 2 sequencer prototype built to measure how transaction ordering policy affects Data Availability (DA) cost and latency on Ethereum. The sequencer submits real EIP-4844 blob transactions to a local Ethereum Deneb devnet and compares two ordering policies: FCFS (First-Come First-Served) and DES (DA Efficiency Score).

The core question: **can a smarter ordering policy reduce the number of blobs a sequencer needs to submit, and at what latency cost?**

---

## 2. Research Questions

- **RQ1:** How much blob waste does FCFS produce under different workloads?
- **RQ2:** How much does DES reduce blob count, and under which conditions?
- **RQ3:** How do the α/β/γ weight parameters affect blob count and latency?

---

## 3. System Architecture

```
[CSV Dataset]
     ↓
[Virtual Clock Simulation]    — advances 60s per tick, admits txs by arrival_ms
     ↓
[Mempool]                     — holds admitted txs, removes after batching
     ↓
[Ordering Engine]             — FCFS or DES
     ↓
[Batcher]                     — greedy 128KB packing + zstd level-3 compression
     ↓
[Blob Sender]                 — real EIP-4844 blob tx via alloy SidecarBuilder
     ↓
[Deneb Devnet]                — Reth + Lighthouse via Kurtosis, 6s slots
     ↓
[Metrics Collector]           — blob count, fill rate, ordering latency, inclusion latency
```

### Key parameters
| Parameter | Value |
|-----------|-------|
| Blob size | 128 KB (131,072 bytes) |
| Batch window | 60 seconds |
| Blob submission delay | 2,000ms (Deneb max 6 blobs/slot) |
| Compression | zstd level 3 |
| DES default weights | α=0.33, β=0.33, γ=0.34 |

## 4. Ordering Policies

### 4.1 FCFS

Sorts all transactions in the current 60-second window by `arrival_ms` (earliest first). Per-sender nonce ordering is enforced. The batcher then greedily packs in that order.

### 4.2 DES (DA Efficiency Score)

DES uses a fit-aware HOL (Head-of-Line) selection loop. For every transaction slot in the output:

1. **Filter:** keep only candidates whose `payload_size + current_batch_size <= 128KB`
2. **Score:** among fitting candidates, compute DES score and pick the highest
3. **If nothing fits:** close the current blob (`current_batch_size = 0`), open a new one, score from scratch

```
DES(tx) = α × wait_score + β × compress_score + γ × fit_score
```

**wait_score** = `(sim_clock_ms - tx.arrival_ms) / batch_timeout_ms`, clamped to [0, 1]
— prioritizes transactions that have been waiting longer (starvation prevention)

**compress_score** = `1.0 - (payload_size / 10_000)`, clamped to [0.1, 1.0]
— prioritizes smaller/more compressible transactions (smaller txs have more ABI zero-padding)

**fit_score** = `payload_size / remaining_blob_space` if tx fits, else `0.0`
— prioritizes transactions that use remaining blob space efficiently

The scores are recomputed on every iteration because `remaining_blob_space` changes as each transaction is placed. `wait_score` and `compress_score` are fixed per transaction per window; `fit_score` is dynamic.

### 4.3 Critical insight: the HOL filter drives blob savings, not the scores

The filter on step 1 is a hard constraint — a transaction that does not fit in the current blob cannot be selected regardless of its score. This is what prevents premature blob boundary crossing.

The α/β/γ weights rank candidates **within the fitting set only**. They have no influence over whether a blob boundary is crossed. As a result:

- All DES weight configurations produce the same blob count
- The weights only control latency tail behavior
- High α (wait_score) prevents starvation, bounding tail latency
- Low/zero α allows large transactions to be indefinitely deferred, causing pathological max latency

## 5. Dataset

### Source
Ethereum mempool data from January 1, 2025 (Flashbots mempool dumps). Contains timestamp, calldata size, 4-byte selector, sender address, and nonce.

### Transaction type classification
| Type | Examples | Compressibility |
|------|----------|-----------------|
| transfer | ERC-20 transfer/approve, WETH wrap | High (~1.55:1) — repetitive ABI zero-padding |
| swap | Uniswap, 1inch, Paraswap | Medium (~1.2:1) — variable path arrays |
| mint | NFT/token mints | Medium |
| other | Contract calls, deployments | Low (~1.09:1) |

### Prepared datasets (5,000 transactions each)
| Dataset | Size filter | Time span | Avg txs/window | Blob overflow |
|---------|-------------|-----------|----------------|---------------|
| small_heavy | 1–300 B | 10 min | ~500 | None — 34 KB/window fits in 1 blob |
| large_heavy | >2,000 B | 10 min | ~500 | Heavy — ~9.6 blobs/window |
| mixed | all sizes | 5 min | ~1,000 | Moderate — ~3.2 blobs/window |

### Typed data generation
Since the CSV only contains metadata, `types.rs` generates synthetic calldata bytes that match the real-world compressibility of each tx type:
- `transfer`: 75% zero density → 4-6:1 zstd ratio
- `mint`: 60% zero density → 3-4:1 zstd ratio
- `swap`: 35% zero density → 2-3:1 zstd ratio
- `other`: 15% zero density → ~1.5:1 zstd ratio

### Devnet
- Execution client: Reth
- Consensus client: Lighthouse
- Orchestration: Kurtosis (`ethereum-package`)
- Deneb enabled from epoch 0, slot time 6 seconds
- Blob submission: alloy `SidecarBuilder<SimpleCoder>`

```bash
# Set devnet credentials
export ETH_RPC_URL="http://127.0.0.1:<port>"
export SENDER_PRIVATE_KEY="<key>"

# Run one experiment
export DATASET="../large_heavy.csv"
export ORDERING_POLICY="des"   # or "fcfs"
cargo run --release

# DES weight tuning
export DES_ALPHA="0.50"
export DES_BETA="0.25"
export DES_GAMMA="0.25"
```

### Full 6-run script
```bash
cargo build --release 2>/dev/null

for DATASET in "../small_heavy.csv" "../large_heavy.csv" "../mixed.csv"; do
    for ORDERING_POLICY in "fcfs" "des"; do
        export DATASET ORDERING_POLICY
        echo "=== $(basename $DATASET .csv) | $ORDERING_POLICY ==="
        cargo run --release 2>/dev/null
    done
done
```

## 7. Results

All six runs completed end-to-end with real EIP-4844 blobs confirmed on-chain on the Reth devnet (May 7, 2026).

### 7.1 Primary comparison: FCFS vs DES across workloads

| Dataset | Policy | Blobs | Uncomp Fill | Ordering Avg | Ordering P95 | Ordering Max | Inclusion Avg |
|---------|--------|-------|-------------|--------------|--------------|--------------|---------------|
| small_heavy | FCFS | 11 | 28.18% | 1,224ms | 7,471ms | 13,736ms | 5,787ms |
| small_heavy | DES | 11 | 28.18% | 1,232ms | 7,527ms | 13,736ms | 5,976ms |
| large_heavy | FCFS | 195 | 87.17% | 655ms | 2,748ms | 3,951ms | 5,978ms |
| large_heavy | DES | **178** | **95.50%** | 1,904ms | 9,143ms | 26,435ms | 5,998ms |
| mixed | FCFS | 20 | 79.15% | 1,349ms | 5,813ms | 19,116ms | 5,622ms |
| mixed | DES | 20 | 79.15% | 1,876ms | 8,028ms | 31,729ms | 5,577ms |

**Ordering latency** = estimated time from tx arrival to when the ordering algorithm selected it within the window. Measured by spreading selection times evenly across the batch window, weighted by position in the ordered output.

**Inclusion latency** = time from blob tx submission to on-chain confirmation. Consistently ~5.7–6.0 seconds across all runs — this is L1 slot behavior (one slot = one blob included), completely independent of ordering policy.

### 7.2 Key findings

**Blob savings only occur on large_heavy (-8.7%).**
DES requires two conditions simultaneously:
1. **Blob overflow** — the window's total tx volume must exceed 128 KB. Without overflow there are no blob boundary decisions to optimize.
2. **Size variance** — tx sizes must vary enough that some txs fit in the current blob tail and others don't. Without variance the HOL filter has no discriminating power.

`small_heavy` fails condition 1: 500 txs × ~68B avg = ~34 KB/window, which fits in one blob.
`mixed` partially fails condition 2: the large number of small txs means blob tails can almost always be filled by arrival order alone.
`large_heavy` satisfies both: 500 txs × ~2,468B avg = ~9.6 blobs/window, with 2,000–5,000B size range creating real fit decisions at blob boundaries.

**Fill rate improves with blob savings.**
large_heavy fill rate: 87.17% (FCFS) → 95.50% (DES), +8.33 percentage points. Tighter packing at blob boundaries means fewer partially-filled blobs.

**DES costs latency on large_heavy.**
- Avg ordering latency: 655ms (FCFS) → 1,904ms (DES), 3× increase
- P95 ordering latency: 2,748ms (FCFS) → 9,143ms (DES), 3.3× increase
- Max ordering latency: 3,951ms (FCFS) → 26,435ms (DES), 6.7× increase

This is the HOL-blocking cost. Large transactions that don't fit in the current blob tail are deferred repeatedly until a new blob opens. The more times a tx is skipped, the more it accumulates wait time.

**Inclusion latency is completely independent of ordering.**
~6 seconds for every run. The devnet processes one blob per slot (6 seconds). This cannot be improved by the sequencer.

### 7.3 Why α/β/γ weights don't affect blob count

From the previous parameter sweep on large_heavy (all configs produced 178 blobs):

| Configuration | α | β | γ | Blobs | P95 Latency | Max Latency |
|---------------|---|---|---|-------|-------------|-------------|
| FCFS baseline | — | — | — | 195 | ~59,000ms | ~62,000ms |
| pure_wait | 1.00 | 0.00 | 0.00 | 178 | ~58,000ms | ~60,000ms |
| pure_compress | 0.00 | 1.00 | 0.00 | 178 | ~80,000ms | ~99,000ms |
| pure_fit | 0.00 | 0.00 | 1.00 | 178 | ~84,000ms | ~101,000ms |
| equal (default) | 0.33 | 0.33 | 0.34 | 178 | ~68,000ms | ~73,000ms |
| wait_heavy | 0.50 | 0.25 | 0.25 | 178 | ~64,000ms | ~67,000ms |
| no_wait | 0.00 | 0.50 | 0.50 | 178 | ~83,000ms | ~101,000ms |

The blob count is fixed at 178 for every DES configuration. This confirms the HOL filter — not the scoring function — is what saves blobs.

The weights are purely latency controls:
- High α → wait_score dominates → long-waiting txs get priority → starvation prevented → lower tail latency
- Low/zero α → large txs can be deferred indefinitely → max latency exceeds the batch window duration
- β and γ have secondary effects; neither saves additional blobs beyond what the filter already achieves

**Best weight configuration:** `wait_heavy` (α=0.5, β=0.25, γ=0.25) — achieves the full 8.7% blob reduction with the lowest tail latency among all DES configurations.

## 8. Architecture Decisions

### Why the HOL filter is separate from scoring
The filter is a hard constraint enforced before scoring. This makes blob savings weight-independent — you always get 178 blobs on large_heavy regardless of α/β/γ. If the filter were removed, blob count would depend on γ (fit_score): high γ would approximate the filter's behavior softly, but high α or β could override it, causing non-fitting txs to be selected and wasting blob tail space.

### Why compression doesn't save blobs on large_heavy
zstd compression is applied after packing. The ordering engine sees uncompressed sizes and packs to 128 KB uncompressed. Large DEX calldata compresses at only 1.09:1, so compression saves ~11KB per 128KB blob — not enough to meaningfully change boundary decisions. The β weight therefore has limited practical impact on this workload.

### Why the average ordering latency can be misleading
DES produces denser blobs (fewer blobs, more txs per blob). Txs that land in early blob positions get low individual latencies. This pulls the average down slightly even when tail latency is much worse. Max and P95 latency are the honest measures for comparing policies.

---

## 9. Conclusions

DES is a conditional improvement over FCFS:

| Condition | DES benefit |
|-----------|-------------|
| Small txs, no overflow | No benefit — identical to FCFS |
| Mixed sizes, moderate overflow | No benefit — small txs fill tails naturally |
| Large txs (>2KB), heavy overflow | 8.7% blob reduction, 3-6× latency cost |

The 8.7% blob reduction on large_heavy translates directly to 8.7% lower blob fee cost for that workload. At current blob base fees this is meaningful but not dramatic. The latency cost (6.7× higher max latency) is real and must be weighed against the cost saving.

DES is most useful for rollups with specialized workloads dominated by large calldata (e.g. ZK proof data, complex contract interactions). For general-purpose L2s with mixed real-world traffic, FCFS and DES produce equivalent blob counts.
